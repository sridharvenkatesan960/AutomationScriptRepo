1. WHAT IS THIS FRAMEWORK?
================================================================================

This is a TEST AUTOMATION FRAMEWORK that automatically tests websites.

Think of it like this:
- Instead of manually clicking buttons on a website to test
- We write instructions in plain English
- Computer reads instructions and tests the website automatically
- Works on multiple browsers (Chrome, Firefox, Edge)
- Can test with different data (emails, passwords, etc.)

TECHNOLOGIES USED:
------------------
- Cucumber: Write tests in plain English (BDD)
- Selenium: Controls web browsers automatically
- JUnit: Runs the tests
- Maven: Manages project and dependencies
- Java: Programming language
- Apache POI: Reads Excel and CSV files

2. HOW TESTS WORK - SIMPLE FLOW
================================================================================

STEP 1: You write a test in plain English
-----------------------------------------------
Example (in Login.feature file):

   Scenario: Login to website
     Given User enters the Url
     Then User logs in using test case "TC001" from CSV file
     And Click on Submit button
     Then Get the page title


STEP 2: Computer reads the test
-----------------------------------------------
- It sees "Given User enters the Url"
- Looks for matching Java code


STEP 3: Java code executes
-----------------------------------------------
- Opens Chrome browser
- Goes to website URL
- Reads email/password from CSV file
- Types email and password
- Clicks submit button
- Checks if login successful


STEP 4: Results generated
-----------------------------------------------
- Creates HTML report showing pass/fail
- Closes browser
- Ready for next test

4. EACH COMPONENT EXPLAINED
================================================================================

COMPONENT 1: FEATURE FILES (FeatureFiles/*.feature)
----------------------------------------------------
Purpose: Write test scenarios in plain English

Example content of Login.feature:
-----------------------------------
Feature: Launch Naukri and search jobs - Data from CSV File

  @login @csvData
  Scenario: Login to Naukri Portal with TC001 from CSV
    Given User enters the Url
    Then User logs in using test case "TC001" from CSV file
    And Click on Submit button
    Then Get the page title

What does this mean?
- "Given" = Setup step (prepare for test)
- "When/Then" = Action step (do something)
- "And" = Additional step
- "@login" = Tag to organize tests


COMPONENT 2: TEST DATA (TestData/*.csv)
----------------------------------------
Purpose: Store test data separately from code

Example content of LoginData.csv:
----------------------------------

TC001,sridharsush@gmail.com,Gmail@96002,Valid login credentials
TC002,invalid@test.com,WrongPass123,Invalid credentials test

Why separate data?
- Easy to add new test cases

COMPONENT 3: STEP DEFINITIONS (src/test/java/StepDefenitions/)
---------------------------------------------------------------
Purpose: Connect English steps to Java code

Example from CSVDataDrivenLoginSteps.java:
-------------------------------------------
@Then("User logs in using test case {string} from CSV file")
public void userLogsInUsingTestCaseFromCSV(String testCaseId) {
    // Step 1: Get web page elements
    loginPageObj = new loginPageObjects(DriverManager.getDriver());

    // Step 2: Read data from CSV file
    Map<String, String> credentials = dataProvider.getLoginCredentialsFromCSV(testCaseId);
    String email = credentials.get("Email");
    String password = credentials.get("Password");

    // Step 3: Display what we're doing
    System.out.println("Reading data from CSV file...");
    System.out.println("Test Case: " + testCaseId);
    System.out.println("Email: " + email);

    // Step 4: Type email and password on webpage
    loginPageObj.getUsernameField().sendKeys(email);
    loginPageObj.getPasswordField().sendKeys(password);
}

What happens here?
1. Method name matches the English step
2. Reads test case ID (like "TC001")
3. Gets email and password from CSV file
4. Types them into webpage


COMPONENT 4: PAGE OBJECTS (src/main/java/webElements/)
-------------------------------------------------------
Purpose: Store webpage element locations

Example from loginPageObjects.java:
------------------------------------
public class loginPageObjects {
    private WebDriver driver;

    public loginPageObjects(WebDriver driver) {
        this.driver = driver;
    }

    // Find username field on webpage
    public WebElement getUsernameField() {
        return driver.findElement(By.id("usernameField"));
    }

    // Find password field on webpage
    public WebElement getPasswordField() {
        return driver.findElement(By.id("passwordField"));
    }

    // Find submit button on webpage
    public WebElement getSubmitButton() {
        return driver.findElement(By.xpath("//button[@type='submit']"));
    }
}

Why do this?
- If webpage changes, update only this file
- Reuse across multiple tests
- Clean and organized


COMPONENT 5: ACTIONS (src/test/java/Actions/)
----------------------------------------------
Purpose: Business logic and reusable actions

Example from loginActions.java:
--------------------------------
public class loginActions {
    private loginPageObjects loginPageObj;

    // Open browser
    public WebDriver launchBrowser() {
        WebDriver driver = commonUtils.launchBrowser();
        loginPageObj = new loginPageObjects(driver);
        return driver;
    }

    // Go to URL
    public void navigateToUrl(String url) {
        commonUtils.navigateToUrl(url);
    }

    // Type username
    public void enterUsername(String username) {
        loginPageObj.getUsernameField().sendKeys(username);
    }

    // Type password
    public void enterPassword(String password) {
        loginPageObj.getPasswordField().sendKeys(password);
    }

    // Click submit button
    public void clickSubmitButton() {
        loginPageObj.getSubmitButton().click();
    }
}

What's the point?
- Reusable actions
- Separates what to do from how to do it
- Easy to maintain


COMPONENT 6: DRIVER MANAGER (src/main/java/Utils/DriverManager.java)
---------------------------------------------------------------------
Purpose: Manage browser lifecycle

Key responsibilities:
- Open browser (Chrome, Firefox, or Edge)
- Close browser
- Manage multiple browsers in parallel
- Thread-safe (each test gets own browser)

How it works:
1. When test starts, creates browser instance
2. Configures browser settings
3. Provides browser to test
4. When test ends, closes browser

Why ThreadLocal?
- Allows parallel test execution
- Each thread has its own browser
- No conflicts between tests


COMPONENT 7: DATA READERS (src/main/java/Utils/)
-------------------------------------------------

CSVReader.java:
- Reads CSV files
- Converts to Java Map objects
- Returns data to tests

ExcelReader.java:
- Reads Excel (.xlsx) files
- Converts to Java Map objects
- Returns data to tests

TestDataProvider.java:
- High-level API for data access
- Automatically detects CSV or Excel
- Provides simple methods to get data


COMPONENT 8: TEST RUNNERS (src/test/java/Runner/)
--------------------------------------------------
Purpose: Configure and execute tests

ChromeRunner.java example:
--------------------------
@RunWith(Cucumber.class)
@CucumberOptions(
    features = "FeatureFiles",              // Where test scenarios are
    glue = {"StepDefenitions", "Hooks"},    // Where Java code is
    plugin = {"pretty", "html:target/cucumber-reports-chrome.html"},
    tags = "@login",                         // Run tests with @login tag
    dryRun = false                          // Actually execute tests
)
public class ChromeRunner {
    @BeforeClass
    public static void setup() {
        System.setProperty("browser", "chrome");  // Use Chrome browser
    }
}

What does each runner do?
- ChromeRunner: Runs tests in Chrome
- FirefoxRunner: Runs tests in Firefox
- EdgeRunner: Runs tests in Edge


COMPONENT 9: HOOKS (src/test/java/Hooks/TestHooks.java)
--------------------------------------------------------
Purpose: Setup and cleanup before/after tests

@Before Hook (runs before each test):
- Logs scenario name
- Can load test data
- Prepare environment

@After Hook (runs after each test):
- Closes browser
- Logs test result (pass/fail)
- Cleanup

Why hooks?
- Automatic setup/cleanup
- No need to remember in each test
- Always runs, even if test fails

7. KEY CONCEPTS SIMPLY EXPLAINED
================================================================================

CONCEPT 1: BDD (Behavior-Driven Development)
---------------------------------------------
What it is:
- Writing tests in plain English
- Anyone can read and understand
- Tests serve as documentation

Example:
Instead of:
    assert(loginPage.usernameField.text == "user@test.com")

We write:
    Given User is on login page
    When User enters "user@test.com"
    Then Username field should contain "user@test.com"

Benefits:
- Business people understand tests
- Less technical knowledge needed
- Clear requirements


CONCEPT 2: Data-Driven Testing
-------------------------------
What it is:
- Separating test data from test logic
- Same test runs with different data
- Data stored in CSV or Excel

Example:
One test, multiple data sets:

Test Code (same):
    Log in with email and password

Test Data (different):
    TC001: user1@test.com, Pass123
    TC002: user2@test.com, Pass456
    TC003: user3@test.com, Pass789

Benefits:
- Easy to add test cases
- No code changes for new data
- Non-programmers can manage data


CONCEPT 3: Page Object Model (POM)
-----------------------------------
What it is:
- Organizing element locators in classes
- One class per webpage
- Reusable methods for actions

Without POM:
    driver.findElement(By.id("username")).sendKeys("test");
    driver.findElement(By.id("username")).clear();
    driver.findElement(By.id("username")).sendKeys("newuser");
    (Repeated in multiple tests)

With POM:
    loginPage.enterUsername("test");
    loginPage.clearUsername();
    loginPage.enterUsername("newuser");
    (Reusable everywhere)

Benefits:
- Single place for element locators
- Easy maintenance
- Reusable code


CONCEPT 4: Parallel Execution
------------------------------
What it is:
- Running multiple tests simultaneously
- Each test in separate browser/thread
- Faster overall execution

Sequential:
    Chrome test ‚Üí 15 seconds
    Firefox test ‚Üí 15 seconds
    Edge test ‚Üí 15 seconds
    Total: 45 seconds

Parallel:
    Chrome test  ‚îê
    Firefox test ‚îú‚îÄ‚Üí All run together ‚Üí 15 seconds
    Edge test    ‚îò
    Total: 15 seconds (60% faster!)

How it works:
- Each runner class runs in separate JVM
- ThreadLocal ensures no conflicts
- Maven Surefire manages parallel execution


CONCEPT 5: Cucumber Tags
-------------------------
What they are:
- Labels to organize and filter tests
- Start with @
- Can have multiple tags

Example:
    @login @smoke @high-priority
    Scenario: Quick login test

    @search @regression @low-priority
    Scenario: Advanced search test

Usage:
    mvn test -Dcucumber.filter.tags="@smoke"        (Run smoke tests only)
    mvn test -Dcucumber.filter.tags="@login and @high-priority"  (AND condition)
    mvn test -Dcucumber.filter.tags="@smoke or @regression"      (OR condition)


CONCEPT 6: Hooks
----------------
What they are:
- Code that runs before/after tests
- Automatic setup and cleanup
- Always executes

Types:
    @Before     - Runs before each scenario
    @After      - Runs after each scenario
    @BeforeAll  - Runs once before all tests
    @AfterAll   - Runs once after all tests

Common uses:
    @Before: Open browser, login, prepare data
    @After:  Close browser, cleanup data, save logs


CONCEPT 7: ThreadLocal
----------------------
What it is:
- Java mechanism for thread-specific data
- Each thread has its own copy
- No sharing between threads

Why needed:
    Without ThreadLocal:
        Thread 1: Opens Chrome ‚îÄ‚îê
        Thread 2: Opens Firefox ‚îº‚îÄ‚Üí Both use same driver ‚Üí Conflict!
        Thread 3: Opens Edge ‚îÄ‚îÄ‚îÄ‚îò

    With ThreadLocal:
        Thread 1: Opens Chrome  ‚Üí Own driver instance
        Thread 2: Opens Firefox ‚Üí Own driver instance
        Thread 3: Opens Edge    ‚Üí Own driver instance
        No conflicts!

8. HOW TO RUN TESTS
================================================================================

PREREQUISITE:
-------------
1. Java 17+ installed
   Check: java -version

2. Maven installed
   Check: mvn -version

3. Browser installed (Chrome recommended)


BASIC COMMANDS:
---------------

1. Run all tests in Chrome:
   mvn test -Dtest=ChromeRunner

2. Run all tests in all browsers (parallel):
   mvn clean test

3. Run specific browser:
   mvn test -Dtest=FirefoxRunner

4. Run with specific tag:
   mvn test -Dtest=ChromeRunner -Dcucumber.filter.tags="@login"

5. Clean and run:
   mvn clean test

6. Skip tests (just compile):
   mvn compile


VIEW REPORTS:
-------------
After tests run, open:
   target/cucumber-reports-chrome.html  (in web browser)

Or command line:
   Linux/Mac: open target/cucumber-reports-chrome.html
   Windows: start target/cucumber-reports-chrome.html


TROUBLESHOOTING:
----------------
If tests fail to run:
1. mvn clean install         (Clean build)
2. Check Java version        (Should be 17+)
3. Check browser installed   (Chrome, Firefox, Edge)
4. Check CSV file exists     (TestData/LoginData.csv)
